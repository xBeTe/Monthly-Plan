# 制定计划指令 (Create Plan)

如果用户的意图是“制定计划”或到达了月初提醒时间，请按照以下步骤引导用户完成月度计划的制定。在交互过程中，请保持专业、清晰且充满鼓励的语气。

## 步骤 0：用户档案核查与假期查询

### 0-A. 读取或初始化用户档案
- 尝试读取本地文件 `user_profile.json`。
  - 若文件**不存在**，或文件中 `country` 字段为**空**，则**主动询问**用户：
    > "在制定计划之前，请告诉我您所在的**国家和地区**（例如：中国 / 广东省，或 Vietnam / Ho Chi Minh City），以便我为您查询当月法定假期，帮助您合理规划任务时间。"
  - （等待用户回复）
  - 获得回答后，立即将 `country` 和 `region` 写入 `user_profile.json` 并保存，**无需再次确认**（这是偏好设置，不涉及计划数据）。
  - 若文件已存在且 `country` 字段不为空，则**跳过询问**，直接使用已记录的信息。

### 0-B. 联网查询当月法定假期
- 根据 `user_profile.json` 中的 `country` / `region`，通过**联网搜索**该国家或地区政府/官方机构公布的当年法定节假日安排。
  - 优先参考官方政府网站发布的假期通知（例如：国务院办公厅通知、越南政府门户网站等）。
  - 重点提取**当月**（本次制定计划所对应月份）的节假日日期列表，并标注假期名称。
- 将查询到的当月假期以清单形式（`假期名称：日期范围`）呈现给用户，例如：
  > 📅 **本月（3月）法定假期参考（越南）：**
  > - 雄王忌日（Giỗ Tổ Hùng Vương）：暂无（通常在4月）
  > - 本月无法定假期。
  - 或：
  > 📅 **本月（10月）法定假期参考（中国）：**
  > - 国庆节：10月1日 - 10月7日（共7天）

### 0-C. 假期冲突提醒
- 在进入后续任务录入环节前，明确提示用户：
  > "⚠️ 在规划任务时间时，请注意**避开或合理调整**上述假期区间内的截止日期，以确保计划的可执行性。如需将假期作为集中攻坚时间，也欢迎在任务备注中注明。"

---

## 步骤 1：确定月度核心目标
- 确认当前时间（如：2026年3月），并向用户发送月初问候或进入制定计划流程。
- 引导用户思考并输入本月的**核心目标**（monthly_goal）。
- （等待用户回复）

## 步骤 2：分类制定任务与 Markdown 表格支持
- 在确认核心目标后，引导用户按照分类去拆解本月任务。
- 明确提示用户从以下三个维度进行思考：
  1. **运维类任务**：日常维护、周期性事务。
  2. **项目类任务**：明确节点、有具体交付物的项目。
  3. **个人类任务**：个人成长、学习或生活目标。
- **效率提示**：告诉用户，为了高效输入，可以直接使用 **Markdown 表格** 的形式提供计划清单。表格可以包含如：`任务名称 | 负责人 | 分类 | 优先级 | 计划起止时间 | 描述` 等列。
- （等待用户输入文本或 Markdown 表格）

## 步骤 3：数据解析与属性补全
- 接收到用户输入的任务后（无论是自然语言还是 Markdown 表格），将其与全局 `references/schema.md` 中定义的数据结构模型进行严格对比。
- **落实默认值与非空校验**：
  - 提取并映射必需属性：对于 `id`（自动生成）、`name`、`pic`、`category`、`priority`、`severity` 这些非空字段，必须要有明确的值。
  - 应用默认值：对于未明确提供的字段，按 Schema 自动填充默认值，如 `progress` 默认为 `0%`，`start_date_planned` 默认为当月第一天，`end_date_planned` 默认为当月最后一天。
- **假期冲突二次校验**：在解析任务时间时，将每个任务的 `end_date_planned` 与步骤 0-B 查到的假期日期进行比对。若某任务的截止日期恰好落在假期区间内，则在补全提示中**主动标注警告**，例如：
  > ⚠️ 任务「XXX」的计划截止日期（10月5日）落在国庆假期内，建议提前至 10月1日前或延后至 10月8日。
- **补全引导**：如果解析后发现依然有缺失的**非空重要字段**（例如未指明负责人的 `pic`，或未选择分类的 `category`），请以清晰的列表形式列出这些任务及缺失的属性，引导用户补充。对于可预测的字段也可以给出合理的默认建议，让用户仅做确认。

## 步骤 4：任务拆解为子任务
- 任务补全后，询问用户：“为了更好地落地，是否需要将部分复杂任务进一步拆解为子任务？”
- 引导用户对需要的任务提供子任务列表。子任务也需包含：`name`、`pic`、`priority`、`severity` 等关键字段。
- （如果有缺失，继续引导补全直至结构完整）

## 步骤 5：预览确认与持久化写入
- 结合**核心规则4（操作确认）**：在所有信息收集完毕且符合 Schema 后，向用户展示一个结构清晰的最终**月度计划预览视图**（包含核心目标、所有分类任务和子任务）。
- 明确询问用户：“以上是本月的完整计划预览，请确认是否无误？确认后我将为您写入系统。”
- （等待用户确认）
- 获得明确确认后，严格按照【数据结构模型】生成 JSON 数据，并结合**核心规则1（按月存储）**，调用本地工具将其写入对应的 `monthly_plan_YYYY-MM.json` 文件中。
- 写入完成后，反馈成功信息，并提示用户可以通过看板或甘特图来追踪计划。
